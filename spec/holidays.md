# 공휴일 데이터 소스 전략

## 기본 전략 (무키)

- 기본 소스: **Nager.Date Public Holidays API**
- 국가코드: `KR`
- 장점
  - API 키 없이 즉시 사용 가능
  - 연도 기준 조회가 단순하고 응답 구조가 안정적
  - 프론트엔드 환경에서 빠르게 통합 가능

## 대안 전략 (키 필요)

- 대안 소스: **data.go.kr 한국천문연구원 특일정보(공휴일) API**
- 장점
  - 국내 공공데이터 기반 운영으로 정책/도메인 신뢰도가 높음
  - 기관 표준 운영 체계와 맞추기 쉬움
- 단점
  - 인증키 발급/관리 필요
  - 호출 제한/운영 정책을 별도 관리해야 함

## 운영 가이드

- 기본 운영은 Nager.Date를 사용한다.
- 아래 상황에서 data.go.kr 대안을 검토한다.
  1. 내부 정책상 공공데이터 소스 사용이 필수인 경우
  2. 특정 연도/특정 공휴일 정확도 검증에서 보정이 필요한 경우
  3. 감사/컴플라이언스 요구로 데이터 출처 고정이 필요한 경우

## 캐싱 전략

- 기본 캐싱: **연도 단위 메모리 캐시** (`Map<year, holidays[]>`)
  - 동일 연도 재조회 시 네트워크 호출 없이 즉시 반환
- 확장 옵션(제안): **localStorage 캐시**
  - 키 예시: `holidays:KR:2026`
  - TTL(예: 7일/30일) 적용 후 만료 시 재조회
  - 장애 시 최근 캐시를 fallback으로 사용 가능

## 실패 대응 원칙

- 네트워크/응답 오류 시 사용자에게 원인과 재시도 안내를 표시한다.
- 부분 연도 조회 실패 시 전체 실패로 처리해 운영자가 상태를 명확히 인지하도록 한다.
- 필요 시 운영 모드에서 수동 공휴일 입력을 병행한다.
